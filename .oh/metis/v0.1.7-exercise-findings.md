## Bugs

### B1: `kind` filter broken for new symbol kinds
**Severity: HIGH**

`search(query: "NodeKind", kind: "enum_variant")` → "No symbols matching". Same for `kind: "macro"` and `kind: "type_alias"`. These kinds exist in `NodeKind` enum and are extracted, but the `kind` filter parameter doesn't match them.

**Root cause hypothesis:** Enum variants are stored with `kind: "field"` in the index, not `kind: "enum_variant"`. Confirmed: `SearchMode`'s variants (Hybrid, Keyword, Semantic) all show as `field` kind. The release notes advertise enum variants as a new kind, but they're not filterable as such.

Similarly, `kind: "macro"` and `kind: "type_alias"` return nothing. The `macro_rules!` definitions exist in test code (src/extract/rust.rs) and type alias support exists in the extractor, but neither kind is findable via the filter.

**Impact:** Agents cannot browse or filter by these new symbol kinds. The release notes promise "eight new symbol kinds" but they're not discoverable through the `kind` parameter.

### B2: Empty query + kind/synthetic filter = error instead of listing
**Severity: MEDIUM**

`search(query: "", kind: "enum_variant")` returns "Empty query. Please describe what you're looking for" instead of listing all symbols of that kind. The empty-query path only works with `sort_by: "importance"` or `sort_by: "complexity"`. Adding `kind`, `synthetic`, or `root: "all"` to an empty query without sort_by produces the error.

**Expected:** `search(query: "", kind: "function")` should list all functions. This is a natural browsing pattern.

### B3: `is_static`, `generics`, `pattern_hint` metadata not surfaced in search output
**Severity: MEDIUM**

The release notes list `is_static`, `generics`, and `pattern_hint` as new metadata fields. These are extracted and stored (confirmed by finding `detect_pattern_hint` function and `is_static` field references in extractors), but they do not appear in `search` output for any symbol. Only `decorators` metadata appears.

**Impact:** Agents can't see whether a method is static, what generics a type has, or what design pattern a class follows — even though this data is computed.

### B4: LSP stuck in "starting..." state
**Severity: LOW (may be environment-specific)**

Throughout the entire test session (~3 minutes), LSP status remained "starting...". This means:
- `tests_for` mode returned no results (needs call edges from LSP)
- Impact analysis only found import-level dependencies, not call-site dependencies
- Graph neighbors only shows `defines` edges, not `calls` edges

The index footer consistently shows: `LSP: starting...`

## Friction

### F1: `search_mode` silently ignored on flat search
The `search_mode` parameter (hybrid/keyword/semantic) is accepted without error on flat search, but the tool description says "Only affects how entry nodes are found when using query + mode; flat search always uses name/signature matching." An agent will naturally try `search(query: "auth handler", search_mode: "semantic")` expecting semantic ranking and get name-matched results instead, with no indication that the parameter was ignored.

**Suggestion:** Either make `search_mode` work for flat search (use hybrid ranking), or return a warning when it's specified without `mode`.

### F2: oh_search_context quality gap between keyword and semantic
- `oh_search_context(query: "BLAKE3 content hashing embedding cache", search_mode: "keyword")` → 0.96 relevance, excellent results
- `oh_search_context(query: "how does the embedding cache work", search_mode: "hybrid")` → 0.46 relevance, returned unrelated "How it works" section from commit-tagging-convention.md

Natural language queries (the primary use case for agents) perform significantly worse than keyword queries. The hybrid default doesn't rescue this.

### F3: Outcome progress suggests deprecated tool names
The `outcome_progress` response footer says: "Use `search_symbols` to explore further, `graph_query` with any ID above for impact/neighbors" — these are deprecated aliases. Should say `search` and `search(mode: ...)`.

### F4: No way to discover valid `kind` values
There's no introspection to find out what `kind` values the filter accepts. An agent has to guess between "enum_variant" vs "variant" vs "enum" vs "field". A help/schema endpoint or listing valid kinds in the tool description would reduce trial-and-error.

### F5: Compact output doesn't show edge types
When using `compact: true` with graph neighbors, the output shows the connected symbols but not what edge type connects them (calls, defines, depends_on). The non-compact output also doesn't explicitly label edge types. Agents need this to understand relationships.

## Observations

- **15,312 symbols indexed** across the codebase — schema v9
- **Synthetic constants dominate** — 4,143 consts out of ~15K symbols are inferred string literals. These add noise to search results (e.g., searching "auth" returns `"author"` string constants)
- **Impact analysis is shallow without LSP** — only 3 LOW-risk import statements found for `agent-alignment` outcome. With LSP call edges, this would be richer
- **Edge-type weighted PageRank** seems effective — `LspTransport` (0.413) and `LangConfig` (0.319) are genuinely central to the architecture
- **repo_map is the standout feature** — one call replaces an entire orientation loop. Best new tool for agent onboarding

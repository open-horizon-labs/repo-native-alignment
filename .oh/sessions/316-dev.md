# Dev Pipeline -- #316 Subsystem metadata and scoped queries
**Issue:** #316
**PR:** (pending)
**Started:** 2026-03-16

## Phase 1: Problem Statement
Subsystem detection computes cluster assignments but only renders them in `repo_map`. The assignments aren't stored on nodes, so no other tool can use them. Agents can see subsystems exist but can't query within or across them.

### Acceptance Criteria
- Every node has `metadata["subsystem"]` after graph build
- `search(query="scan", subsystem="scanner")` returns only scanner-subsystem symbols
- `search(mode="impact", node="X")` output groups results by subsystem with entry points
- Subsystem metadata survives LanceDB persist/reload
- Existing tests pass, new tests for subsystem filtering

## Phase 2: Solution Space

### Analysis
Three changes, building on each other:

1. **Store cluster_id as node metadata (~20 lines)**: After `detect_communities()`, write `metadata["subsystem"] = cluster_name` on each node. The Subsystem struct currently doesn't return member node IDs, so we need to add a `member_ids: Vec<String>` field. Then in `build_full_graph_inner`, after step 6 (PageRank), call detect_communities and write subsystem names to node metadata before persisting to LanceDB.

2. **Subsystem-scoped search (~10 lines)**: Add `subsystem` parameter to `Search` struct in tools.rs. Pass it through to `SearchParams`. In `search_flat`, filter nodes where `metadata["subsystem"] == param`.

3. **Subsystem-aware impact (~50 lines)**: After `search_traversal` collects results, group by `metadata["subsystem"]`. Show which subsystems are affected and through which interface function.

### Selected Approach: Direct metadata injection at graph build time

**Why:** The subsystem detection already runs in `repo_map` -- we move it to graph build time so all nodes get their subsystem label. The label persists through LanceDB (metadata is already serialized). Search filtering is a simple metadata check.

# Dev Pipeline — CLI and MCP must share a single service layer
**Issue:** #241
**PR:** (pending)
**Started:** 2026-03-14

## Phase 1: Problem Statement
CLI subcommands and MCP tool handlers are separate code paths. When capabilities are added to MCP tools (e.g., include_artifacts, include_markdown, artifact_types on search), they don't appear in the CLI. This is a design problem: the architecture allows capability drift by default.

**Acceptance Criteria:**
- [ ] A shared service layer exists for search, graph traversal, scan, and stats
- [ ] CLI subcommands call the service layer, not the subsystems directly
- [ ] MCP tool handlers call the service layer, not the subsystems directly
- [ ] Adding a new parameter to the service layer makes it available in both interfaces
- [ ] CLI `search` has feature parity with MCP `search` (include_artifacts, include_markdown, artifact_types, search_mode, graph traversal modes)

## Phase 2: Solution Space

### Analysis

The current code has two separate implementations:
1. **CLI** (`main.rs` lines 292-361): Custom inline search with only basic filters (query, kind, language, file, limit). No include_artifacts, include_markdown, artifact_types, search_mode, compact, sort_by, min_complexity, etc.
2. **MCP** (`handlers.rs`): Full-featured search via `handle_search()` on `RnaHandler`, which requires `&self` (embed_index, lsp_status, graph lock, repo_root, etc.)

### Selected Approach: Service Functions Module

**Level:** Local optimum

Create `src/service.rs` with service functions that take explicit parameters (GraphState ref, optional EmbeddingIndex, repo_root, etc.) instead of requiring `&RnaHandler`. Both CLI and MCP handlers become thin dispatchers.

For search specifically:
- Extract a `SearchParams` struct (plain data, no MCP dependencies)
- The MCP `Search` struct converts to `SearchParams`
- CLI `SearchArgs` converts to `SearchParams`
- `service::search()` takes `SearchParams` + `&GraphState` + optional embed/markdown context

For graph: similar pattern with `GraphParams` + `service::graph_query()`
For stats: `service::stats()` takes `&GraphState` + repo_root
For scan: thin wrapper already exists (build_full_graph), minimal change needed

### Why not the alternatives?
- **Band-aid** (just add missing CLI flags): Doesn't prevent future drift
- **Redesign** (trait-based interface layer): Over-engineered for 4 commands
- **Reframe** (make CLI call MCP over stdio): Adds latency and complexity

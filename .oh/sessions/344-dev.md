# Dev Pipeline -- #344 Background LSP edge parity

## Phase 1: Issue Analysis

**Problem:** MCP server background LSP enrichment produces ~4,347 edges vs CLI `scan --full` foreground producing ~8,410 edges. Nearly half the LSP edges are missing in the background path.

## Phase 2: Root Cause

Two issues found in `src/server/enrichment.rs`:

### 1. Missing fallback in `spawn_background_enrichment`

`spawn_background_enrichment` (called from `build_full_graph_inner(true)` during MCP server startup) spawns a tokio task that reads graph state via `bg_graph.read().await`. If the graph state is `None` (possible race: the graph hasn't been stored by the caller yet), the task **silently returns with 0 LSP edges**:

```rust
// OLD: silent return when graph is None
if let Some(ref gs) = *guard {
    (gs.nodes.clone(), gs.index.clone())
} else {
    bg_lsp_status.set_complete(0);
    return;  // <-- loses all LSP enrichment
}
```

Meanwhile, `spawn_lsp_enrichment` (the other background path) had a proper fallback to use the passed-in nodes:

```rust
// spawn_lsp_enrichment's pattern (already correct)
} else {
    (bg_nodes, GraphIndex::new())  // <-- fallback
}
```

### 2. Incremental persist after background enrichment

After background LSP enrichment, the code used `persist_graph_incremental` to write only the new LSP edges. The initial `build_full_graph_inner` persist (line 600) already wrote tree-sitter-only edges via `persist_graph_to_lance` (DROP + CREATE).

While the incremental persist *should* add LSP edges on top of tree-sitter edges, using a full persist after enrichment is more robust and matches the foreground pipeline pattern. The foreground path does: extract -> LSP -> full persist (all edges together).

## Phase 3: Fix

Changes to `src/server/enrichment.rs`:

1. **`spawn_background_enrichment`**: Added fallback to `bg_nodes` when graph state is `None`, matching `spawn_lsp_enrichment`'s pattern. Added diagnostic logging for node count.

2. **Both `spawn_background_enrichment` and `spawn_lsp_enrichment`**: Changed from `persist_graph_incremental` to `persist_graph_to_lance` (full persist) after LSP enrichment completes. This writes the complete graph (tree-sitter + LSP edges) atomically, ensuring the LanceDB cache has all edges for the next server startup.

3. **Added diagnostic logging**: Both functions now log the node count being enriched, making future discrepancies easier to diagnose.

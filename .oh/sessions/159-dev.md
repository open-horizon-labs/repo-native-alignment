# Dev Pipeline — Static vs instance method distinction in Function metadata
**Issue:** #159
**PR:** (filled in Phase 3)
**Started:** 2026-03-13

## Phase 1: Problem Statement
Issue #159 already has a clear problem statement with acceptance criteria. Agents cannot distinguish constructors, static/associated functions, and instance methods because all Function nodes lack method-type metadata. The fix is adding `metadata["is_static"]` to Function nodes inside class/struct/impl scope.

**Acceptance Criteria (from issue):**
- Function nodes have `metadata["is_static"]` = "true"/"false" when inside class/struct/impl scope
- Rust: functions without `self` param in impl blocks marked static
- Python: `@staticmethod`/`@classmethod` detected, or no `self`/`cls` first param
- Java/C#/TS: `static` keyword detected in function text
- Go: add `is_static: "true"` when no receiver for consistency
- Top-level functions do NOT get the field
- Tests for each supported language

## Phase 2: Solution Space
**Selected:** Generic extractor enhancement (local optimum level)

The generic extractor already has `parent_scope` tracking and parameter inspection. Add `is_static` detection at the point where Function nodes are emitted, using language-specific heuristics:

1. **Generic (Java/C#/TS):** Check if function body text contains "static" keyword AND has parent_scope
2. **Rust:** Check first parameter for self/&self/&mut self in impl blocks (parent_scope exists)
3. **Python:** Check first parameter name (self/cls) and decorators (@staticmethod/@classmethod)
4. **Go:** Post-process in go.rs specials -- add is_static based on receiver presence

This approach modifies `collect_nodes` in generic.rs minimally and uses per-language specials for edge cases.

# Dev Pipeline -- #198 Stale worktree roots persist in LanceDB
**Issue:** #198
**PR:** #201
**Started:** 2026-03-13

## Phase 1: Problem Statement

When git worktrees are removed from disk, their symbols, embeddings, and metadata persist in LanceDB indefinitely. Every search returns 3-4x duplicate results from dead roots, inflating symbol count from ~7K to 25K.

**Root cause:** `build_full_graph` discovers live roots via `with_worktrees()` but never compares them against roots already stored in LanceDB. The background scanner has stale-root detection (`removed_slugs` diff), but this only catches removals that happen while the server is running. On restart, stale data is loaded verbatim.

**Acceptance Criteria:**
- [x] At startup, roots stored in LanceDB but absent from disk are pruned from all tables (symbols, edges, file_index, pr_merges)
- [x] `delete_nodes_for_roots` also cleans file_index and pr_merges (currently only handles symbols + edges)
- [x] No data for live roots is deleted
- [x] Tests verify stale root pruning logic

## Phase 2: Solution Space

Single viable approach: eager pruning at startup. The infrastructure already exists -- `delete_nodes_for_roots` handles the actual deletion, `with_worktrees` discovers live roots. The missing piece was comparing live vs stored roots in `build_full_graph`.

No alternative approaches needed -- this is a straightforward lifecycle hook addition.

## Phase 3: Execute

Implementation in `src/server.rs`:
1. Added `get_stored_root_ids()` -- queries LanceDB symbols table for distinct root_id values
2. Expanded `delete_nodes_for_roots()` -- now cleans all 4 root_id-bearing tables (was only symbols+edges)
3. Added stale root pruning in `build_full_graph()` -- after root discovery, before scanning, diffs live vs stored roots
4. Added 2 tests verifying the pruning logic

All 665 tests pass, no new clippy warnings.

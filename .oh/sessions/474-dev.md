## Phase 1: Problem Statement

**Issue:** #474 ‚Äî perf: parallelize per-file extraction with rayon ‚Äî Extractor trait is already Send+Sync

**Problem:** Per-file tree-sitter extraction is sequential but embarrassingly parallel. The `Extractor` trait is `Send + Sync`. `extract_scan_result` uses a sequential `for` loop. On a 10-core M4, a 500-file scan takes ~10s sequentially but could complete in ~1s with `rayon::par_iter`.

**Acceptance Criteria:**
- [ ] `rayon` added as dependency in `Cargo.toml`
- [ ] `extract_scan_result` uses `par_iter` for the file processing loop
- [ ] All tests pass (`cargo test`)
- [ ] No regressions: extracted nodes/edges are identical to sequential output (order may differ)
- [ ] Before/after timing reported on a medium-sized scan

## Phase 2: Solution Space

**Candidates:**
1. **par_iter on extract_scan_result** (local optimum) ‚Äî Add `rayon` dep, collect `files_to_process` into a parallel iterator, merge results. The `ExtractionResult` merge is the only coordination point. This is what the issue asks for. Low risk.
2. **par_iter + threshold skip** ‚Äî Same as above but skip rayon for small file counts (<10) to avoid thread overhead. Micro-optimization, probably not needed.
3. **Async worker pool with backpressure** ‚Äî More complex, enables future streaming. Overkill for this.

**Selected:** Option 1 ‚Äî `par_iter` on `extract_scan_result`. The `Extractor` trait is already `Send + Sync`, and each file is independent. The only change needed: collect per-file results in parallel then fold into one `ExtractionResult`.

**Key consideration:** The current loop logs per-file timing. Parallel execution means per-file log messages will be interleaved. That's acceptable ‚Äî `tracing` is already thread-safe.

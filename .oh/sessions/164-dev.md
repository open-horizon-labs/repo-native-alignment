# Dev Pipeline -- #164 Configurable Pattern Hint Detection
**Issue:** #164
**PR:** (pending)
**Started:** 2026-03-13T14:00:00Z

## Phase 1: Problem Statement

PR #184 shipped hardcoded naming convention pattern detection (`detect_pattern_hint` with 15 suffixes). The patterns are baked into the binary -- users cannot customize them. Different teams have different architectural conventions (e.g., `*Gateway`, `*Interactor`, `*UseCase`) that RNA currently ignores. Users also cannot disable false-positive-prone defaults for their codebase.

### Acceptance Criteria
- [ ] Pattern definitions loaded from `.oh/config.toml` under a `[patterns]` section
- [ ] Built-in defaults (the 15 from PR #184) apply when no config exists
- [ ] Users can add custom suffix/hint pairs
- [ ] Users can disable specific built-in patterns
- [ ] `detect_pattern_hint` uses the merged (defaults + overrides) pattern list
- [ ] Documentation in config.toml example and in code comments
- [ ] Existing tests continue to pass; new tests cover config loading

## Phase 2: Solution Space

### Candidates

**A. Thread config through call chain (local optimum)**
- Add `PatternConfig` field to `GenericExtractor`, thread through `collect_nodes` -> `detect_pattern_hint`
- Requires updating `GenericExtractor::new` signature, all 14 language extractors, 60+ test call sites
- Clean but extremely invasive for a config feature

**B. OnceLock global pattern config (band-aid/local optimum)**
- `PatternConfig` stored in `std::sync::OnceLock`, initialized at server startup from `.oh/config.toml`
- `detect_pattern_hint` reads from the OnceLock; falls back to hardcoded defaults if uninitialized
- Zero changes to call chain; only `detect_pattern_hint`, config loading, and server init change
- Trade-off: global state, but this IS global config -- it doesn't vary per-file or per-language

**C. Trait-based config injection (redesign)**
- Add config to the `Extractor` trait itself via an associated type or method
- Most architecturally pure but massive refactor for marginal benefit

**Selected: B (OnceLock)**
- Reason: The pattern config is genuinely global (same patterns for all languages), loaded once at startup, immutable after init. A `OnceLock` is the idiomatic Rust solution for this. Zero invasion on the 60+ call sites that create `GenericExtractor`.
- Level: Band-aid / local optimum hybrid

### Config shape
```toml
[patterns]
# Additional suffix -> hint mappings (merged with built-in defaults)
extra = [
  ["gateway", "gateway"],
  ["interactor", "interactor"],
  ["usecase", "use_case"],
]
# Disable specific built-in patterns by hint name
disable = ["manager", "service"]
```

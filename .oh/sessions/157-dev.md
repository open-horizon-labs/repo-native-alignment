# Dev Pipeline -- #157 Decorators/attributes not captured as metadata on decorated symbols
**Issue:** #157
**PR:** (filled in Phase 3)
**Started:** 2026-03-13

## Phase 1: Problem Statement
Issue #157 has a clear problem statement with acceptance criteria. Agents cannot filter or discover symbols by their behavioral annotations because the generic extractor discards decorator/attribute nodes. In framework codebases, the decorator IS the primary semantic signal.

**Acceptance Criteria:**
- Decorators/attributes captured in node metadata (e.g., `metadata["decorators"] = "@app.route(\"/api\"), @login_required"`)
- Works for Python decorators, Rust attributes, Java annotations at minimum
- `search_symbols` output shows decorators when present
- Agents can grep/filter by decorator name in search results

## Phase 2: Solution Space
**Selected approach:** Add `decorator_node_kinds` field to `LangConfig`, collect decorators as metadata in `collect_nodes`.

This is a local optimum approach -- minimal blast radius (only `generic.rs`, `configs.rs`, language extractors, `store.rs`, `server.rs`), leverages existing `metadata` BTreeMap, and follows the established pattern for other metadata (storage, cyclomatic, etc.).

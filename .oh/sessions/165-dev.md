# Dev Pipeline — Hybrid search: combine sparse keyword + dense vector scoring

**Issue:** #165
**PR:** (pending)
**Started:** 2026-03-13

## Phase 1: Problem Statement

RNA has two disjoint search paths: sparse name/signature matching and dense vector search. Dense is bad at exact keywords, sparse is bad at semantic similarity. Hybrid gives both.

### Acceptance Criteria
- [ ] `search` tool flat mode uses hybrid scoring (sparse + dense)
- [ ] LanceDB FTS index created alongside vector index at embed time
- [ ] Results rank better than pure vector for keyword queries (e.g., exact function names)
- [ ] Results rank better than pure keyword for semantic queries (e.g., "error handling")
- [ ] No regression in search latency
- [ ] Tests

## Phase 2: Solution Space

### Selected: Option A — LanceDB native hybrid search

LanceDB 0.26.2 natively supports hybrid search combining FTS (tantivy BM25) + vector with RRF score fusion. The API:

1. Create FTS index: `table.create_index(&["title", "body"], Index::FTS(Default::default())).execute().await`
2. Hybrid query: combine `full_text_search(FullTextSearchQuery::new(query))` + `nearest_to(embedding)` on the same query - LanceDB auto-detects and runs `execute_hybrid` with RRF reranking.

**Why Option A over Option B (app-level fusion):**
- LanceDB handles RRF internally, eliminating score normalization complexity
- Tantivy FTS is already a transitive dependency
- Minimal code change - enhance existing `search()` method

### Implementation Plan
1. Add `lance-index` direct dep for `FullTextSearchQuery`
2. In `index_all_inner()`, after table creation, create FTS index on `title` + `body` columns
3. In `search()`, add `full_text_search` to the vector query to trigger hybrid mode
4. Expose `search_mode` param: "hybrid" (default), "keyword", "semantic"
5. Tests

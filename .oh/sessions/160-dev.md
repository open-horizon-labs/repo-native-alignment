# Dev Pipeline -- Trait/interface method signatures not indexed

**Issue:** #160
**PR:** (filled in Phase 2)
**Started:** 2026-03-13

## Phase 1: Problem Statement

Agents searching for API contract methods via `search_symbols` find nothing because RNA indexes trait/interface declarations as opaque single nodes, silently discarding the individual method signatures that define the contract. Trait methods define what things can do -- they are the architectural skeleton of a codebase -- yet they are the one category of functions RNA drops entirely from symbol search and graph queries.

### Root Cause
- Rust `scope_parent_kinds` missing `"trait_item"` -- only has `impl_item`, `struct_item`, `enum_item`
- Signature-only method node types not in `node_kinds`: Rust `function_signature_item`, TS `method_signature`, Go `method_spec`
- Same pattern affects all languages with interfaces/protocols

### Acceptance Criteria
- [ ] Trait/interface method signatures indexed as Function nodes
- [ ] Methods have `parent_scope` metadata pointing to trait/interface name
- [ ] `search_symbols("serve")` finds `fn serve(&self)` from `trait Service { fn serve(&self); }`
- [ ] Works for Rust traits, TypeScript interfaces, Java interfaces at minimum
- [ ] Default method implementations also captured
- [ ] No duplicate nodes when both trait method and impl method exist

## Phase 2: Solution Space

### Selected: Config-level fix (band-aid / local optimum)

The fix follows the existing LangConfig pattern exactly. No new abstractions needed.

**Rust:** Add `"trait_item"` to `scope_parent_kinds`, add `"function_signature_item"` to `node_kinds`
**TypeScript:** Add `"interface_declaration"` to `scope_parent_kinds`, add `"method_signature"` to `node_kinds`
**Java:** Add `"interface_declaration"` to `scope_parent_kinds` (methods already use `method_declaration`)
**Go:** Custom code needed -- Go interfaces are handled in `go.rs` `extract_type_spec`, need to extract `method_spec` children
**C#:** Add `"interface_declaration"` to `scope_parent_kinds` (methods already use `method_declaration`)
**Swift:** Add `"protocol_declaration"` to `scope_parent_kinds`

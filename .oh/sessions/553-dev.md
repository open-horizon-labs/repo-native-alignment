# Issue 553: Framework passes as FrameworkDetected consumers

## Problem

Three passes ran as registry passes with self-gating instead of being proper
`FrameworkDetected` consumers:

1. `fastapi_router_prefix_pass` — ran unconditionally in `EnrichmentFinalizer` Step 2
   before framework detection. Self-gated only via `files_to_scan.is_empty()` after O(N)
   collection. On non-FastAPI repos: N ApiEndpoint node scans for nothing.

2. `sdk_path_inference_pass` — ran unconditionally in `EnrichmentFinalizer` Step 2b.
   Had a TS/JS const pre-check (added in #551) but still invoked on every repo.

3. `nextjs_routing_pass` — had `detected_frameworks.contains("nextjs-app-router")` gate
   but ALSO a `has_ts_js` fallback that caused it to run on any repo with TypeScript/
   JavaScript nodes — even plain React apps with no Next.js.

Root cause: `framework_detection_pass` ran at Step 9, AFTER the FastAPI passes (Steps 2
and 2b). So those passes could not gate on framework detection results.

## Solution

Move `framework_detection_pass` to Step 2 (immediately after the merge/patch step).
This makes detected frameworks available for all subsequent pass gates.

**New pass ordering in `EnrichmentFinalizer::run_passes`:**
1. Merge tree-sitter + LSP nodes/edges and apply metadata patches
2. `framework_detection_pass` — NOW RUNS FIRST (only needs Import nodes)
3. `fastapi_router_prefix_pass` — gated on `detected_frameworks.contains("fastapi")`
4. `sdk_path_inference_pass` — gated on `detected_frameworks.contains("fastapi")`
5. `api_link_pass`
6. `openapi_sdk_link_pass`
7. `manifest_pass`
8. `tested_by_pass`
9. `import_calls_pass`
10. `directory_module_pass`
11. Framework-gated passes: pubsub, websocket, nextjs_routing, grpc, extractor_config

The `nextjs_routing` gate is now pure `detected_frameworks.contains("nextjs-app-router")`.
The `has_ts_js` fallback is dropped: plain TS/JS repos without Next.js produce no
page-route `ApiEndpoint` nodes.

## New consumer: SdkPathInferenceConsumer

Added `SdkPathInferenceConsumer` that subscribes to `FrameworkDetected("fastapi")`.
Same pattern as `FastapiRouterPrefixConsumer` — monitoring/signal slot.

Registered in `build_builtin_bus()`. Re-exported from `consumers/framework/sdk.rs`.

## Test fixes

Two integration tests (`test_lsp_only_root_api_endpoint_nodes_survive_rescan`,
`test_removed_lsp_only_root_nodes_get_pruned`) had fixture files with no `next/`
imports. Without imports, `framework_detection_pass` never fires
`FrameworkDetected("nextjs-app-router")`, so `nextjs_routing_pass` is not invoked.

Fix: added `pages/_app.tsx` with `import type { AppProps } from 'next/app'` to each
test fixture. This is a realistic Next.js project shape and makes framework detection
fire correctly.

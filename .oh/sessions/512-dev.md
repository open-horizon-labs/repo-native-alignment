# Dev Pipeline — feat: optional repo parameter for MCP tools

**Issue:** #512
**PR:** #513
**Started:** 2026-03-22

## Phase 1: Problem Statement

RNA MCP server starts with a fixed `--repo` path. Agents in git worktrees can't use MCP tools to query their own worktree's graph — the server always returns results from the main repo. This forces agents to fall back to grep/read.

The server already scans worktrees via `with_worktrees()` and includes them in the graph as separate roots. The gap is that agents have no way to request results from a specific repo path via MCP.

**Acceptance Criteria:**
- `search`, `repo_map`, `outcome_progress` tools each accept `repo: Option<String>`
- When `repo` is provided, load graph from that path's LanceDB
- When `repo` is absent, behavior unchanged
- Agent instructions updated to show `repo` parameter usage
- Scan < 120s performance gate

## Phase 2: Solution Space

### Approach A (Recommended): Load separate GraphState from repo path

When `repo` is provided:
1. Canonicalize the path
2. Call `load_graph_from_lance(&repo_path)` for a fresh `GraphState`
3. For embeddings: open `EmbeddingIndex::new(&repo_path)` if the artifacts.lance table exists
4. Use `root_filter = None` (show all nodes — no slug filtering needed)

**Why this approach:**
- Clean and direct: exactly matches what the agent wants
- No need to know the root slug
- `load_graph_from_lance` already exists and works
- Consistent with how the server loads its own graph
- Graceful degradation: if path has no LanceDB, return helpful error

**Trade-offs:**
- Each call with `repo` loads the graph on-demand (no caching); acceptable for agent use
- Only works for already-scanned repos (agent must have run `repo-native-alignment scan --repo . --full` first)
- The `embed_index` for semantic search won't be available unless `EmbeddingIndex::new` succeeds

### Changes Required

**`src/server/tools.rs`:** Add `repo: Option<String>` to `Search`, `RepoMap`, `OutcomeProgress` structs (after existing `root` field). Keep description short (<80 chars).

**`src/server/handlers.rs`:** In `handle_search`, `handle_repo_map`, `handle_outcome_progress` — detect `repo` param, load fresh `GraphState` if provided, use it instead of `self.graph`.

**`src/service/mod.rs`:** `SearchParams::from_mcp_search` does not need changes — repo is handled in the handler layer before the service call.

**Agent instructions:**
- `.claude/agents/dev-pipeline.md`: Add note showing agents can pass `repo` to scope queries to their worktree
- `.claude/agents/ship.md`: Same addition
